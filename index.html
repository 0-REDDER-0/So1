BLOQUE 01

Análisis Arquitectónico y
Gestión de Recursos

Diagnóstico de cuellos de botella en servidores Ubuntu Server y mecanismos de protección de memoria.

Contexto

Ralentización del Servidor Central

El servidor central de renderizado (Ubuntu Server) se ha ralentizado drásticamente porque varios diseñadores lanzaron scripts automáticos de transcodificación de video 4K y análisis de metadatos pesados a la vez. El equipo de TI nota que el sistema está al límite.

1. Clasificación de la Infraestructura

Ante la saturación del servidor por tareas simultáneas de transcodificación 4K, los componentes se categorizan de la siguiente manera:

Hardware Software (Apps) Recursos Lógicos
• Memoria RAM: Almacenamiento volátil de trabajo.
• Procesador (CPU): Cómputo bruto multihilo.
• Disco NVMe: I/O de alta velocidad para assets. 		• Compilador Python: Intérprete que ejecuta los scripts de automatización.
• Base de Datos: Instancia PostgreSQL que almacena los metadatos. 		• Demonios (Systemd): Gestión de procesos del ciclo de vida del servidor.
• Puertos de Red: Sockets abiertos para transferencia de ingesta multimedia.

Contexto

Aislamiento multiusuario

Los editores de video temen que los scripts pesados de un usuario corrompan los renders de animación de otro en la misma sesión.

2. Aislamiento de Memoria Virtual

El Rol de la MMU

Para evitar la corrupción cruzada entre renders paralelos de distintos diseñadores, el procesador implementa aislamiento por hardware:

  • MMU (Memory Management Unit): Traduce las direcciones de memoria virtual del proceso a direcciones físicas reales.
  • Protección de Memoria: El Kernel restringe que un proceso acceda al espacio asignado a otro.

Consecuencias de Fallo

Segmentation Fault: Terminación inmediata del proceso infractor por violación de direccionamiento.

Data Leakage: Riesgo crítico de fuga de datos confidenciales entre proyectos de clientes en el mismo espacio físico.

BLOQUE 02

Ecosistemas y
Modelos de Negocio

Toma de decisiones de TI: Evaluación de dependencias, costos de licenciamiento y abstracción multiplataforma.

Contexto

Debate del Ecosistema

La gerencia del estudio debate qué ecosistema usar para reducir costos sin perder la potencia que exigen los editores de video y diseñadores 3D para llevar a cabo sus proyectos sin interrupciones.

3. Matriz de Decisión Estratégica

Dimensión Sistemas Abiertos (Ubuntu Server) Sistemas Propietarios (macOS)
Dependencia de Proveedor Nula. Modificaciones internas sin restricciones del fabricante (Vendor Lock-in libre). Alta. Apple controla rígidamente el ciclo de vida del software y la compatibilidad de hardware.
Auditoría de Código Total. Permite revisar el código fuente línea por línea para mitigar filtraciones antes del estreno. Cerrado. Secreto comercial corporativo; se depende de los parches oficiales del fabricante.
Costo Total (TCO) $0 en licencias. El presupuesto se optimiza e invierte en hardware NVMe y soporte técnico. Elevado. Alta inversión requerida para estaciones de trabajo premium de nivel Mac Studio / Mac Pro.

Contexto

Factor Multiplataforma

Los diseñadores afirman: "Como usamos un entorno multiplataforma, no importa que nuestras estaciones de edición sean Mac con procesadores Apple Silicon (arquitectura ARM) y nuestro servidor de renderizado sea x86_64".

4. El Factor Multiplataforma

Convivencia ARM y x86_64

Los editores usan estaciones Mac (Apple Silicon - ARM) integradas con un servidor de renderizado Linux (Intel/AMD - x86_64) sin fricciones de compatibilidad gracias a dos capas clave:

1. System Calls (Llamadas al Sistema)

El SO expone una interfaz abstracta idéntica para el software, aislando los binarios de la arquitectura del procesador.

2. Drivers como Traductores

Los controladores traducen de manera nativa estas peticiones de software abstracto a las señales eléctricas específicas de cada procesador.

BLOQUE 03

Inmersión en la
Arquitectura y CLI

Flujo de ejecución interna de comandos, administración de almacenamiento, monitoreo y auditoría de permisos.

Contexto

Falla Crítica en la Consola

Hay un error al intentar guardar un proyecto final desde la consola de comandos de un editor y el administrador de TI debe auditar rápidamente el flujo del sistema para solucionar el colapso.

5. Viaje de un Comando en el Sistema (`ls`)

1. Shell

Interpreta la sintaxis del administrador y localiza el binario ejecutable en el PATH.

2. Kernel

Recibe la petición vía System Calls (fork() y execve()) para iniciar el hilo.

3. Drivers

Traducen la llamada a bajo nivel y leen físicamente los bloques de memoria del disco NVMe.

4. Salida

El Kernel formatea los bytes crudos y el Shell renderiza la lista de archivos legible.

6. Mitigación de Crisis en Consola

Almacenamiento

El volumen multimedia reporta llenado súbito.

df -h

Muestra espacio libre/usado en formato legible (Gigabytes/Terabytes).

Rendimiento

Identificación del script colgado de transcodificación.

top / htop

Monitoreo interactivo en tiempo real de carga de CPU, RAM y PIDs.

Finalización

Forzar el cierre inmediato del proceso crítico bloqueado.

kill -9 4502

Envía señal SIGKILL que el Kernel ejecuta de inmediato omitiendo limpieza intermedia.

Contexto

Seguridad

  • Almacenamiento: El disco de videos parece lleno. Ejecutamos df -h (Comando base: df, Flag: -h para formato legible en GB/TB).
  • Rendimiento: El script colgado del usuario es el PID 4502

    • 7. Auditoría de Permisos de Seguridad

    Análisis de la máscara de protección del archivo crítico del servidor de renderizado:

    - rwx r-x r--   1   root   root   config.conf

    Dueño (root)

    rwx

    Control absoluto. Lectura, escritura/modificación y ejecución de scripts.

    Grupo (TI/Diseño)

    r-x

    Lectura de configuración y ejecución del renderizador, imposibilitado para alterar el archivo maestro.

    Otros (Externos)

    r--

    Restricción estricta de solo lectura. Evita manipulaciones maliciosas o accidentales desde fuera de la red local.
